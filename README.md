# Askforge

A local document + web question-answering app built with LangChain: it loads
your documents, embeds them into a **Chroma vector store**, and uses an LLM
(**OpenAI, Mistral, or Google Gemini** — your choice) to answer questions
grounded in that content (retrieval-augmented generation / RAG).

Available as a CLI (`app.py`) and a web chat UI (`app_web.py`).

## Features
- Load `.txt` documents from `docs/`
- Split and embed them into a **local, persisted Chroma vector store** (`chroma_db/`)
- Ask questions in plain language; answers are generated by the LLM using
  the most relevant retrieved chunks — not a raw document dump
- **Live web search** (via DuckDuckGo, no API key needed) supplements the
  documents — the LLM prefers your documents when they have the answer, and
  falls back to the web for general knowledge, current events, or anything
  your documents don't cover. Works even with an empty `docs/` folder.
- Shows which source(s) — document filenames and/or web links — each answer
  was grounded in
- **Three supported LLM providers** (OpenAI, Mistral, or Google Gemini) —
  pick whichever you have a working key for; switching is a one-line `.env`
  change
- **Upload files from the web UI** (the "+" button next to the chat input,
  like ChatGPT's attach button):
  - **Documents** (`.txt`, `.md`, `.pdf`, `.docx`, `.csv`) are parsed, chunked,
    and added straight into your knowledge base — no restart needed.
  - **Images** (`.png`, `.jpg`, `.jpeg`, `.webp`, `.gif`) aren't added to the
    knowledge base (there's nothing to embed) — instead, attach one and ask
    a question about it directly; it's sent to a vision-capable call on
    whichever provider you're using for that one question. Image support
    quality varies by provider/model — if one doesn't handle it well, switch
    providers via `RAG_PROVIDER` in `.env`.

## Setup
1. Create a Python 3.11 or 3.12 virtual environment (avoid your global/system
   Python — see the note at the bottom on why):
   ```bash
   python3 -m venv venv
   source venv/bin/activate   # Windows: venv\Scripts\activate
   ```
2. Install dependencies:
   ```bash
   pip install -r requirements.txt
   ```
3. Copy `.env.example` to `.env` and add ONE API key:
   ```bash
   # OpenAI - needs billing/credits set up, no free tier
   # (get one at https://platform.openai.com/api-keys)
   OPENAI_API_KEY=your_key_here

   # OR Mistral AI - genuinely free "Experiment" tier, no billing required
   # (get one at https://console.mistral.ai/api-keys)
   MISTRAL_API_KEY=your_key_here

   # OR Google Gemini
   # (get one at https://aistudio.google.com/app/apikey)
   GOOGLE_API_KEY=your_key_here
   ```
   Whichever key is present is the provider that gets used. If you set more
   than one, priority is OpenAI > Mistral > Google — or force a specific one
   with `RAG_PROVIDER=openai`, `RAG_PROVIDER=mistral`, or `RAG_PROVIDER=google`
   in `.env`.
4. Add `.txt` files to `docs/`
5. Run the CLI version:
   ```bash
   python app.py
   ```
   ...or the web version:
   ```bash
   python app_web.py
   ```
   then open http://localhost:5000

The first run builds the Chroma index from your documents (this calls the
embeddings API, so it needs your API key and an internet connection). Later
runs reuse the persisted index in `chroma_db/` unless you delete that folder
or call `build_or_load_vectorstore(force_rebuild=True)`. Each provider gets
its own collection inside `chroma_db/`, so switching providers never mixes
incompatible embedding vectors.

## Project structure
```
core.py             ← shared RAG pipeline (loading, splitting, embeddings, Chroma, LLM QA)
app.py              ← CLI front end
app_web.py          ← Flask web front end
templates/index.html← chat UI
docs/               ← put your .txt files here
chroma_db/          ← persisted vector store (created on first run, gitignored)
```

## Deploying (e.g. Railway)

1. **Set your API key(s) as environment variables in the hosting dashboard**,
   not in a file — `.env` is gitignored on purpose and never gets pushed to
   GitHub, so nothing in it reaches your host automatically. On Railway:
   your project → **Variables** tab → add `OPENAI_API_KEY` (or `MISTRAL_API_KEY`
   / `GOOGLE_API_KEY`) with the real value. Without this, the app starts but
   answers with a "no API key found" error.
2. **The included `Procfile`** runs the app with `gunicorn`, a real production
   WSGI server, instead of Flask's built-in dev server (which prints its own
   warning telling you not to use it in production - now you're not). It also
   removes any ambiguity about which file to run: this project has two entry
   points (`app.py` for the CLI, `app_web.py` for the web server), and without
   an explicit `Procfile`, auto-detection can guess wrong and try to run the
   CLI version, which has no HTTP server at all and fails with "Application
   failed to respond".
3. **Port/host binding is automatic** - the app detects the platform's `PORT`
   environment variable (which Railway sets for you) and binds to `0.0.0.0`
   on that port automatically. You don't need to configure this yourself.
4. **Persistent storage (important, easy to miss)**: container filesystems on
   Railway (and most hosts) are **ephemeral by default** - anything written
   to disk, including `chroma_db/` and files added through the "+" upload
   button, is wiped on every redeploy or restart. To keep your index and
   uploads permanently:
   - In Railway: your service → **Settings → Volumes → New Volume**, mount it
     at e.g. `/data`
   - Add two more environment variables: `DOCS_DIR=/data/docs` and
     `CHROMA_DIR=/data/chroma_db`
   - Redeploy. From then on, uploads and the vector index survive restarts.
   Skipping this isn't broken, exactly - the app rebuilds its index from
   whatever's in the repo's `docs/` folder on every restart - but anything
   uploaded through the UI will quietly disappear on the next deploy.
5. Redeploy after adding the variables, and check the **Deploy Logs** tab if
   it still doesn't respond - that shows the real Python traceback, which is
   far more specific than the generic "Application failed to respond" page.

## Notes
- `app_web.py` binds to `127.0.0.1` and runs with debug mode **off** by
  default. Only set `FLASK_HOST=0.0.0.0` / `FLASK_DEBUG=true` if you
  understand the security tradeoffs of exposing the Werkzeug debugger or
  your local network.
- Never commit your `.env` file — it's already listed in `.gitignore`. If a
  key is ever exposed (e.g. shared in a zip, repo, or screenshot), rotate it
  immediately in the relevant provider's dashboard
  ([OpenAI](https://platform.openai.com/api-keys),
  [Mistral](https://console.mistral.ai/api-keys),
  [Google AI Studio](https://aistudio.google.com/app/apikey)).
- **Why a dedicated venv matters**: this project intentionally avoids heavy
  ML packages like `torch`/`sentence-transformers`. If your global Python
  already has them installed (common after other ML projects), some LangChain
  submodules probe for them at import time, and a broken/incompatible install
  (e.g. torch not yet supporting a brand-new Python version) can crash import
  before our code even runs. A clean venv avoids that entirely.
- **On the OpenAI option**: the API is pay-as-you-go with no free tier — a
  brand-new account or one with no billing set up will hit `insufficient_quota`
  (HTTP 429) immediately. Add a payment method at
  https://platform.openai.com/settings/organization/billing, or use Mistral
  instead if you'd rather not.
- **On the Google Gemini option**: as of mid-2026 Google is migrating API keys
  from the old `AIza...` format to a new `AQ....` "auth key" format, and the
  new format currently has a known, unresolved bug on Google's side
  (`ACCESS_TOKEN_TYPE_UNSUPPORTED`) affecting some accounts. If Gemini gives
  you that error, switch to OpenAI or Mistral instead of troubleshooting
  further — it's a platform-side issue, not something fixable in this code.
- **On web search**: it uses the unofficial `ddgs` package to query DuckDuckGo,
  so no API key is needed, but it isn't a guaranteed/official API — DuckDuckGo
  may occasionally rate-limit or return no results, especially from cloud IPs.
  When that happens the app silently falls back to documents-only rather than
  erroring out. Set `ENABLE_WEB_SEARCH=false` in `.env` to disable it entirely.
